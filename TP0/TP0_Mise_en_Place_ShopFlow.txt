Présentation de ShopFlow
ShopFlow est une API e-commerce simplifiée que vous allez construire pas à pas. Elle gère trois grandes
fonctionnalités : un catalogue de produits, un panier par utilisateur, et un système de commandes. Un
système de coupons de réduction sera ajouté en TP4 via le cycle TDD.
Ce que fait ShopFlow
• GET /products/ : liste tous les produits du catalogue (avec filtre catégorie, prix, pagination)
• POST /products/ : ajouter un produit au catalogue (admin)
• POST /cart/?user_id=X : ajouter un produit au panier d'un utilisateur
• GET /cart/{user_id} : afficher le contenu du panier avec le total TTC
• POST /orders/ : convertir le panier en commande (vérifie le stock, calcule le total, vide le panier)
• PATCH /orders/{id}/status : faire évoluer le statut : pending → confirmed → shipped
• POST /coupons/apply : appliquer un coupon de réduction sur un prix
• GET /health : smoke test, vérifie que l'API est opérationnelle

Relations entre les tables
• Un Cart appartient à un user_id (utilisateur), 1 panier par utilisateur
• Un CartItem relie un Cart à un Product avec une quantité (table de liaison)
• Un Order est créé depuis un Cart, il contient les OrderItems (snapshot du panier)
• Un Coupon peut être appliqué à une Order via coupon_code (FK nullable)
• Les ForeignKey avec ondelete='CASCADE' : si on supprime un Cart, ses CartItems sont supprimés
automatiquement

# app/services/pricing.py
from typing import Optional, List, Tuple
from app.models import Product, Coupon
TVA_RATE = 0.20 # Taux TVA France (20%)
def calcul_prix_ttc(prix_ht: float) -> float:
"""Calcule le prix TTC à partir du prix HT. Lève ValueError si prix <
0."""
if prix_ht < 0:
raise ValueError(f"Prix HT invalide : {prix_ht}")
return round(prix_ht * (1 + TVA_RATE), 2)
def appliquer_coupon(prix: float, coupon: Coupon) -> float:
"""Applique une réduction. Lève ValueError si coupon inactif ou réduction
invalide."""
if not coupon.actif:
raise ValueError(f"Coupon inactif : {coupon.code}")
if not 0 < coupon.reduction <= 100:
raise ValueError(f"Réduction invalide : {coupon.reduction}")
return round(prix * (1 - coupon.reduction / 100), 2)
def calculer_total(
produits: List[Tuple[Product, int]],
coupon: Optional[Coupon] = None
) -> float:
if not produits:
return 0.0 # liste vide → 0
total_ht = sum(p.price * q for p, q in produits) # somme HT
total_ttc = calcul_prix_ttc(total_ht) # → TTC
if coupon:
total_ttc = appliquer_coupon(total_ttc, coupon) # réduction
return total_ttc

# app/services/stock.py
import logging
12from sqlalchemy.orm import Session
from app.models import Product
from app.cache import redis_client
logger = logging.getLogger(__name__)
def verifier_stock(product: Product, quantite: int) -> bool:
"""Retourne True si stock >= quantite, False sinon."""
if quantite <= 0:
raise ValueError(f"Quantité invalide : {quantite}")
return product.stock >= quantite
def liberer_stock(product: Product, quantite: int, session: Session) ->
Product:
"""Libère du stock (annulation) et met à jour le cache Redis."""
if quantite <= 0:
raise ValueError(f"Quantité invalide : {quantite}")
product.stock += quantite
session.commit()
session.refresh(product)
redis_client.set(f"product:{product.id}:stock", product.stock)
return product
def reserver_stock(product: Product, quantite: int, session: Session) ->
Product:
if not verifier_stock(product, quantite):
raise ValueError(
f"Stock insuffisant pour '{product.name}' : "
f"{product.stock} disponible(s), {quantite} demandé(s)."
)
product.stock -= quantite
session.commit()
session.refresh(product)
cache_key = f"product:{product.id}:stock"
redis_client.delete(cache_key) # invalider le cache
logger.info(f"Stock réservé : {product.id}, qty={quantite}")
return product

4.3. Quel est l'avantage de séparer la Les services peuvent être testés sans démarrer FastAPI, sans
logique métier (services/) des routes HTTP, sans TestClient. On importe directement la fonction et
(routes/) du point de vue des tests ? on l'appelle avec des arguments Python.
Donnez un exemple concret avec Exemple avec reserver_stock() : en test unitaire (TP1), on
reserver_stock().

directement avec un product SQLite en mémoire et un

d'intégration (TP2), on passe par l'endpoint POST /orders/

Si reserver_stock() échoue en test unitaire, le bug est localisé
dans le service. Si le test d'intégration échoue mais pas le test
unitaire, le bug est dans la route ou le schéma, diagnostic plus
rapide.
