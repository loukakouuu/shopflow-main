2.6 Stage Coverage
Stage Coverage
stage('Coverage') {
steps {
sh '''
6pytest tests/ \
--cov=app \
--cov-report=xml:coverage.xml \
--cov-report=html:htmlcov \
--cov-report=term-missing \
--cov-fail-under=80 \
--junitxml=junit-report.xml
'''
}
post {
always {
publishHTML(target: [
allowMissing: false,
alwaysLinkToLastBuild: true,
keepAll: true,
reportDir: 'htmlcov',
reportFiles: 'index.html',
reportName: 'Coverage Report'
])
}
failure {
echo 'Coverage < 80% — ajouter des tests'
}
}
}
Fail Fast : ordre des stages
• L'ordre Lint → Unit Tests → Integration Tests → Coverage est volontaire : Fail Fast.
• Lint d'abord : erreur PEP8 détectée en < 10s sans perdre du temps à lancer pytest.
• Tests unitaires avant intégration : les tests rapides (< 1s) signalent les bugs avant les tests lents.
• Coverage en dernier : génère coverage.xml dont SonarQube a besoin au stage suivant.
• Si Lint échoue, les stages suivants ne s'exécutent pas, économie de ressources CI.

3.1 Stage Static Analysis : Pylint + Bandit
Ajoutez ce stage après Coverage dans votre Jenkinsfile :
Stage Static Analysis
stage('Static Analysis') {
steps {
sh '''
# Pylint — code smells et conventions
pylint app/ \
--output-format=parseable \
--exit-zero \
> pylint-report.txt || true
echo "Pylint terminé — voir pylint-report.txt"
# Bandit — analyse sécurité Python (CWE/OWASP)
bandit -r app/ \
-f json \
-o bandit-report.json \
--exit-zero
# Échouer si vulnérabilités HIGH détectées
python3 -c "
import json, sys
data = json.load(open('bandit-report.json'))
high = [r for r in data.get('results',[])
if r['issue_severity'] == 'HIGH']
if high:
print(f'BANDIT: {len(high)} vuln HIGH détectée(s)')
sys.exit(1)
print('BANDIT: aucune vulnérabilité HIGH')
"
'''
}
}
3.2 Stage SonarQube Analysis
Stage SonarQube Analysis
stage('SonarQube Analysis') {

-Dsonar.projectKey=shopflow \
-Dsonar.sources=app \
-Dsonar.tests=tests \
-Dsonar.python.coverage.reportPaths=coverage.xml \
-Dsonar.python.pylint.reportPaths=pylint-report.txt

3.3 Stage Quality Gate
Stage Quality Gate
stage('Quality Gate') {
steps {
timeout(time: 5, unit: 'MINUTES') {
waitForQualityGate abortPipeline: true
// ↑ bloque si :
// - coverage < 80%
// - bugs > 0
// - vulnerabilities > 0
// - code smells > seuil
}
}
}
Points de vigilance SonarQube
• --network host : obligatoire pour que sonar-scanner-cli atteigne sonarqube:9000 depuis un
conteneur Docker.
• coverage.xml doit être généré par le stage Coverage AVANT ce stage : l'ordre des stages est crucial.
• sonar.qualitygate.wait=true dans sonar-project.properties force waitForQualityGate à attendre le
résultat.
• En cas de timeout (Quality Gate pending) : augmenter timeout(time: 10) ou vérifier les logs
SonarQube.
• Si le Quality Gate est configuré trop strictement, ajuster dans SonarQube → Quality Gates →
Conditions.

9Partie 4 : Déploiement Staging
Ajoutez les deux derniers stages : build de l'image Docker taguée avec le SHA Git, et déploiement
sur l'environnement staging (uniquement sur la branche main).
4.1 Stage Build Docker
Stage Build Docker
stage('Build Docker') {
steps {
script {
// TAG = SHA Git court = traçabilité parfaite
env.IMAGE_TAG = sh(
script: 'git rev-parse --short HEAD',
returnStdout: true
).trim()
echo "Build image shopflow:${env.IMAGE_TAG}"
sh "docker build -t shopflow:${env.IMAGE_TAG} ."
}
}
}
4.2 Stage Deploy Staging
Créez d'abord le fichier docker-compose.staging.yml :
docker-compose.staging.yml
# docker-compose.staging.yml
version: '3.9'
services:
api:
image: shopflow:${IMAGE_TAG:-latest} # image buildée au stage précédent
container_name: shopflow-staging
ports:
- "8001:8000" # port staging différent du port dev
environment:
- DATABASE_URL=sqlite:///./shopflow_staging.db
- REDIS_URL=redis://redis-staging:6379
depends_on:
redis-staging:
condition: service_healthy
restart: unless-stopped
redis-staging:
image: redis:7-alpine
container_name: shopflow-redis-staging
healthcheck:
test: ["CMD","redis-cli","ping"]
interval: 5s
timeout: 3s
retries: 5
volumes:
- redis_staging:/data
volumes:
redis_staging:
Stage Deploy Staging
stage('Deploy Staging') {
when {
branch 'main' // uniquement sur la branche main
}
steps {
sh '''
10export IMAGE_TAG=${IMAGE_TAG}
docker compose -f docker-compose.staging.yml pull || true
docker compose -f docker-compose.staging.yml up -d --remove-orphans
docker compose -f docker-compose.staging.yml ps
# Smoke test sur staging
sleep 5
curl -f http://localhost:8001/health || exit 1
echo "Staging déployé et opérationnel"
'''
}
post {
failure {
sh 'docker compose -f docker-compose.staging.yml logs --tail=50'
}
}
}

5.2 Améliorer le bloc post{}
Mettez à jour le bloc post{} du Jenkinsfile pour archiver les artefacts et nettoyer :
Bloc post{} complet
post {
always {
// Archiver les rapports pour consultation dans Jenkins
archiveArtifacts artifacts: 'junit-*.xml,coverage.xml,bandit-report.json',
allowEmptyArchive: true
// Nettoyer les conteneurs Docker dangling
sh 'docker system prune -f --filter label=stage=ci || true'
}
success {
echo "Pipeline OK — ShopFlow:${env.IMAGE_TAG} déployé"
}
failure {
echo "Pipeline FAILED — voir les logs ci-dessus"
// En production : envoyer un email ou une notification Slack
// mail to:'equipe@shopflow.io', subject:"Pipeline FAILED #${env.BUILD_NUMBER}"
}
unstable {
echo 'Pipeline instable — des tests ont échoué'
}
}
