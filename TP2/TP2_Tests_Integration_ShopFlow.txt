TestClient · SQLite · Fixtures · Faker · JUnit XML
Objectif du TP
À l'issue de ce TP, vous saurez écrire des tests d'intégration avec TestClient FastAPI, configurer une BDD
SQLite isolée via les fixtures conftest.py, gérer les données de test avec des fixtures et faker, valider les

Stack pytest · httpx · TestClient FastAPI · SQLite :memory: · faker · pytest-mock
Livrable Suite tests/integration/ + conftest.py complet + rapport JUnit XML
Prérequis TP0 (ShopFlow fonctionnel) + TP1 (fixtures SQLite maîtrisées)
Tests Unitaires vs Tests d'Intégration
TP1 : Tests Unitaires TP2 : Tests d'Intégration
Teste une fonction isolée Teste un endpoint HTTP complet
Mocks pour toutes les dépendances BDD SQLite réelle (pas de mock)
Pas de FastAPI ni de HTTP TestClient simule des requêtes HTTP
assert calcul_prix_ttc(100) == 120 assert response.status_code == 201
Très rapide (< 1ms par test) Plus lent (10-50ms par test)
Couvre la logique métier Couvre routes + services + BDD
Pourquoi les deux types de tests sont nécessaires ?
• Les tests unitaires valident la logique métier en isolation : calcul de prix, gestion du stock, coupons.
• Les tests d'intégration valident que les couches fonctionnent ensemble : la route appelle bien le
service, le service lit bien la BDD, la réponse est bien sérialisée par Pydantic.
• Un bug peut être invisible en test unitaire (les mocks masquent les problèmes d'intégration) mais
détecté en test d'intégration.
• Exemple : calcul_prix_ttc() est correcte (TP1), mais la route /orders/ n'appelle pas calcul_prix_ttc()
(bug détecté uniquement en TP2).
1Partie 1 : Configuration TestClient & Fixtures
1.1 TestClient FastAPI
FastAPI fournit un TestClient basé sur httpx qui simule des requêtes HTTP en mémoire, sans démarrer un vrai
serveur. L'avantage crucial : on peut remplacer la dépendance get_db par une session de test via
dependency_overrides.
Principe du TestClient
# Principe du TestClient avec override de dépendance
from fastapi.testclient import TestClient
from app.main import app
from app.database import get_db
# 1. Créer une BDD de test isolée
engine_test = create_engine('sqlite:///:memory:', ...)
# 2. Override : remplace get_db par notre session de test
def override_get_db():
session = SessionTest()
try:
yield session
finally:
session.close()
app.dependency_overrides[get_db] = override_get_db
# ↑ FastAPI injectera override_get_db() dans toutes les routes
# 3. Créer le client
client = TestClient(app)
# 4. À la fin, nettoyer
app.dependency_overrides.clear()

11Partie 5 : Rapport JUnit XML & Smoke Tests
5.1 Générer un rapport JUnit XML
Jenkins consomme des rapports JUnit XML pour afficher les résultats de tests dans son interface. Générez ce
rapport avec l'option --junitxml :
Terminal
# Lancer les tests d'intégration avec rapport JUnit XML
pytest tests/integration/ -v \
--junitxml=junit-integration.xml \
--no-cov # pas de coverage ici
# Lancer tous les tests avec coverage + JUnit
pytest tests/ \
--junitxml=junit-report.xml \
--cov=app \
--cov-report=xml:coverage.xml \
--cov-report=html \
--cov-fail-under=80
# Vérifier le contenu du rapport XML
cat junit-integration.xml | head -30
Structure d'un rapport JUnit XML
• <testsuites> : racine du document, contient les statistiques globales (tests, failures, errors, time)
• <testsuite name='tests/integration/test_products_api.py'> : un bloc par fichier de test
• <testcase classname='TestListProducts' name='test_liste_vide_au_demarrage'> : un bloc par test
• <failure message='...'> : présent uniquement si le test a échoué - Jenkins affiche ce message
• Jenkins : stage junit 'junit-report.xml' pour publier le rapport dans l'interface
